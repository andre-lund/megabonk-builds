# Plan: Leaderboard empirical-meta layer

## Intent

Bring `megabonk.leaderboard.gg`'s aggregate run data (per-character weapon/tome/item usage over 638 video-verified top runs, patch 1.0.65) into the app as an empirical-meta signal, and wire it into scoring, suggestions, bans, and Generate so recommendations reflect what strong players actually run — not just editorial wiki synergies. `starting.ts` starter derivation and any per-run loadout UI are out of scope.

## Approach

Per [ADR-0001](../../adr/0001-leaderboard-empirical-meta.md), keep the source distinct from wiki data. `scripts/ingest-leaderboard.mjs` fetches a leaderboard page, decodes the inline Nuxt `__NUXT_DATA__` devalue payload, reconciles ids to our canonical names via a small alias table, derives (a) per-character usage counts and (b) thresholded global within-kind co-occurrence edges (lift ≥ 1.25, support ≥ 8), and writes `src/data/leaderboard.json`. `src/lib/meta.ts` turns that into a `MetaIndex`: a sample-shrunk per-character usage fraction and an undirected discovered-co-occurrence adjacency. Scoring gains an optional `meta` param and a `metaBonus` component (usage prior over picks + discovered within-kind pairs); `suggestFor` threads it through, so marginal-gain ranking reflects it. `bans.ts` gains an optional `meta` param adding a usage-aware keep term. `App.tsx` builds the index once and passes it to all three.

## Decision log

- **Empirical co-occurrence is a separate score component, not merged into the wiki synergy adjacency** — keeps the wiki graph's meaning and UI display unchanged, and keeps the two signals independently tunable (per ADR-0001). It reuses no `synergyIndex` edges.
- **Zero-usage bans were rejected as vacuous.** Analysis of the 638 runs: after aliasing, essentially every item/weapon/tome in our data appears in ≥1 top run (only Silver Tome is genuinely never used). A binary "never used → ban" rule would fire on almost nothing, so bans instead use a *graded* usage keep-term (`META_KEEP × usageFrac`): proven items are pushed above the ban cutoff, genuinely low-usage filler still falls through.
- **Co-occurrence is lift, not raw support.** `dexecutioner+katana` share 335 runs at lift 1.07 (both just universally popular); the real discovered synergies are the high-*lift* pairs (axe+flamewalker 1.76, shotgun+sniper 1.54, bloodmagic+shotgun 1.39). Edges are thresholded at lift ≥ 1.25 **and** support ≥ 8 at ingest time; the thresholds are baked into the committed data and documented here.
- **Per-character usage is shrunk toward the global prior** by sample size (`frac = (count + K·globalFrac)/(runs + K)`, K = 20 ≈ a mid-sized character's run count). High-n characters (fox 177, dicehead 161) are barely moved; low-n characters (ninja 9) — where every appearing item reads as 1.0 — are pulled toward the pooled prior. Without a character selected, the global fraction is used.
- **Scoring weights are tunable defaults, not tuned:** `USAGE_POINTS`, `COOC_POINTS`, `META_KEEP`. `metaBonus` is included in `total` (and therefore in tier), on the view that a proven meta build legitimately scores higher; tiers were not re-baselined.
- **Genuine data gaps recorded, not force-mapped:** `scythe` (a leaderboard weapon absent from our wiki data) and `cryptkey` (ambiguous vs. our existing "Key") are left unmatched and logged rather than aliased, to avoid double-counting.
- **Reconciliation aliases** (leaderboard id → our name): character `astronat → Spaceman`; weapons `mine → Mines`, `sniper → Sniper Rifle`; tomes by `id + "tome"` with `hp → Health Tome`; items `pot → Pot (Stainless Steel)`.

### Adversarial review outcome (8 findings, triaged)

- **Fixed — co-occurrence pair key.** The dedupe key joined/split endpoints on a separator that had been written as an invisible NUL byte (`\x00`) in source. It functioned correctly (verified: re-running produced byte-identical edges), but read as a plain space — it fooled the review into a false HIGH "multi-word names corrupt" report. Replaced with an explicit `\t` key **and** the endpoints stored on the entry object, so there is no encode/split round-trip at all.
- **Fixed — ingest robustness.** `c.counts?.[kind] ?? []` now guards the usage loop like the adjacent `c.builds` access, so a character with a partial payload no longer crashes the whole ingest. The untrusted `version` payload key is validated against `/^[\w.\-]{1,32}$/` (fail-loud), keeping junk out of the committed JSON and doubling as a mis-parse check.
- **Accepted risks (documented, not guarded).** The devalue resolver maps unused negative special indices (HOLE/±Infinity/−0) to `NaN` — latent; those never occur on the resolved counts/builds path. `cooccurrence()` drops unreconciled loadout ids without adding them to `unmatched` — never fires, since builds ids are a subset of counts ids (already recorded). Unmatched slug ids and numeric fields are written without charset/`Number()` coercion — the ingest is developer-run and diff-reviewed, values render through React (escaped) and feed math where a bad value surfaces as a visible `NaN`; per minimum-code, no speculative validation added.
- **Generate consistency (T6).** The review flagged that Generate would optimize the meta-less score while the displayed score and gain badges include `metaBonus`. Rather than leave that divergence, `meta` was threaded into `generateBuild` (it already calls `suggestFor`), so Generate optimizes the same metric the UI shows. ADR-0001's deferral was narrowed to `starting.ts` + per-run UI accordingly.
